## Technical Details
- **#47 cut parity by construction.** Rather than reimplement single-host CSG, `extract_mesh_one` resolves guid→step via the indexer's `product_guid`/`product_step_id` arrays, builds the tessellation set as `{target} ∪ {openings voiding target}` from `voids_opening`/`voids_host`, tessellates that subset, and routes it through the **same** `CrossProductCut::route`/`flush` the batch `extract_meshes` uses. Identical operands in → identical fold out. Flush passes `None` for the prism table (forces the manifold fold; geometrically equivalent to the prism fast-path, which is a batch-throughput optimisation only).
- **#47 coordinate decision.** A single product can't span the f32-overflowing extent a whole georeferenced model can, so `m.mesh()` returns `vertices` as **float64 absolute world metres** computed as `(local + mesh_anchor) * unit_scale` — full precision, no `global_shift` bookkeeping for the caller. Batch `meshes()` keeps shifted float32 + `MeshList.global_shift`. This is the one deliberate divergence from "same as meshes()".
- **Validation.** Compared `m.mesh(guid)` against the matching product from `m.meshes()` on G55_RIB + Duplex_A, uncut and cut: faces identical, coords within f32 quantization (~1e-6 m; the f64 getter is strictly more precise). A host wall went 12→48 faces under cut, matching batch. A directly-targeted opening returns `None` in cut mode (suppressed cutter) but the mesh uncut. Unknown GUID → `None`. 227 Rust + 177 Python tests pass.
