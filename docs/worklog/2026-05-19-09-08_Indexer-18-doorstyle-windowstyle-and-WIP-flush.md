## Technical Details
**Why the suffix check was load-bearing on IFC4 and broken on IFC2x3.** The dispatch map intentionally doesn't enumerate every IfcXxxType subclass — schemas add new ones (IfcEarthworksFillType, IfcRoadType, etc.) and an explicit list ages poorly. The fallback `t.len() > 7 && t.starts_with(b"IFC") && t.ends_with(b"TYPE")` lets new schema additions auto-classify as TypeObject without churn. The IFC2x3 schema, however, predates the systematic *Type renaming: door/window typing is named `IfcDoorStyle` / `IfcWindowStyle` (collapsed into IfcDoorType / IfcWindowType in IFC4). They inherit from `IfcTypeProduct` → `IfcTypeObject` → `IfcRoot`, so the field shape (`arg[0]` GlobalId, `arg[2]` Name) matches the existing TypeObject branch — the only thing the indexer needed was a way to *recognise* them. Two `t == b"IFC*STYLE"` byte equality checks add no measurable cost on the hot path (still one memcmp per un-dispatched record) and surface the two names cleanly. A purely "schema-supertype" approach would have required either an embedded subclass table (defeats the suffix-fallback purpose) or a runtime schema lookup (too heavy for the indexer pass).

**Validation flow + cache-staleness gotcha.** First `python3 -c` validation after `maturin develop --release` showed `ifctype = 61` unchanged — the rebuilt `_core.abi3.so` was loading fine but the model was reading from `~/.cache/ifcfast/027b2a6e185668e0/index.parquet`, which is keyed off content hash (size + head 4MB + tail 4MB SHA256). Since the IFC file itself hadn't changed, the cache key was identical and pointed at the parquets written by the previous indexer build. Manual `gio trash` of the cache dir forced a cold parse; result jumped to 99/268 immediately — matches the ifcopenshell `IfcRelDefinesByType` walk count exactly. Worth remembering: any indexer change that affects what gets extracted requires cache eviction for prior-cached files, or bumping `CACHE_VERSION` in `python/ifcfast/cache.py`. I did not bump CACHE_VERSION because the cached schema didn't change — but any user with an existing 027b2a... cache will get stale 61/268 results until they evict. Trade-off: a CACHE_VERSION bump would force every existing user to re-parse all their files, which is more aggressive than this single-bug case warrants.
